**Anthropic Engineering — "Quantifying infrastructure noise in agentic coding evals"**  
https://www.anthropic.com/engineering/infrastructure-noise

Agentic coding benchmarks like SWE-bench and Terminal-Bench are treated as precise measurements of model capability, but infrastructure configuration alone can produce differences exceeding those margins. On Terminal-Bench 2.0, the gap between most- and least-resourced setups was 6 percentage points (p < 0.01).

SWE-bench 和 Terminal-Bench 等 Agentic Coding 基准测试被用来精确比较前沿模型的软件工程能力，但基础设施配置的差异本身就能产生超过那些分数差距的影响。在 Terminal-Bench 2.0 上，资源配置最多和最少的设置之间相差 6 个百分点（p < 0.01）。

The core issue: container runtimes enforce resources via two separate parameters — a guaranteed allocation and a hard kill threshold. When these are set equal (pinned), there's zero headroom for transient spikes, causing spurious OOM kills. The benchmark's official leaderboard uses a more lenient sandboxing provider that allows temporary overallocation.

核心问题：容器运行时有两种独立的资源执行参数——保证分配值（guaranteed allocation）和硬终止阈值（hard kill threshold）。当两者设为相同值（固定）时，对瞬时尖峰没有容错空间，导致虚假 OOM 杀死。官方排行榜使用的沙盒提供程序更宽松，允许临时超额配置。

Running Terminal-Bench across six resource configurations from strict (1x) to uncapped, success rates increased monotonically. Infra error rates dropped from 5.8% (strict) to 0.5% (uncapped). Up to 3x, additional resources fix infrastructure reliability. Above 3x, resources actively help the agent solve harder problems. The recommendation: specify both guaranteed allocation and kill threshold separately per task, with a 3x ceiling as a reasonable default.

在严格（1x）到无限制的六种资源配置下运行 Terminal-Bench，成功率随资源增加而单调上升。基础设施错误率从 5.8%（严格）降至 0.5%（无限制）。在 3x 以内，额外资源解决的是基础设施可靠性问题；超过 3x 后，资源开始实际帮助 Agent 解决更难的问题。建议：以 3 倍作为合理上限，对每项任务分别指定保证分配值和终止阈值。

**Anthropic Engineering — "Claude Code auto mode: a safer way to skip permissions"**  
https://www.anthropic.com/engineering/claude-code-auto-mode

Auto mode is a new Claude Code mode that delegates approvals to model-based classifiers — a middle ground between manual review and `--dangerously-skip-permissions`. Two layers: a server-side prompt-injection probe at input, and a transcript classifier at output.

Auto mode 是 Claude Code 的新模式，将审批委托给基于模型的分类器——介于人工审核和 `--dangerously-skip-permissions` 之间的中间地带。两层防护：输入层的服务器端提示注入探测，输出层的转录分类器。

The transcript classifier runs on Sonnet 4.6 in two stages: a fast single-token filter (errs toward blocking), followed by chain-of-thought reasoning only if flagged. It sees only user messages and agent tool calls — assistant text and tool outputs are stripped. Stage 1 has 8.5% FPR and 6.6% FNR; the full pipeline reduces FPR to 0.4% at the cost of 17% FNR on real overeager actions.

转录分类器在 Sonnet 4.6 上分两阶段运行：第一阶段是快速单 token 过滤器（倾向于阻止），只有被标记才进入第二阶段进行思维链推理。它只看用户消息和 Agent 的工具调用——助手文本和工具输出被剥离。第一阶段 FPR 8.5%、FNR 6.6%；完整流程将 FPR 降至 0.4%，代价是真实过度热心行为上的 FNR 为 17%。

Auto mode is designed for four threat types: overeager behavior (agent exceeds user intent), honest mistakes (agent misunderstands blast radius), prompt injection, and model misalignment. The default block rules cover destroy/exfiltrate, degrade security posture, cross trust boundaries, and bypass review/affect others. The 17% miss rate on real overeager actions reflects cases where the classifier correctly identifies danger but can't determine if a prior approval covers this specific action's blast radius.

Auto mode 针对四类威胁：过度热心行为（Agent 超出用户意图）、无心之失（Agent 误解影响范围）、提示注入和模型不对齐。默认阻止规则涵盖销毁/泄露、降低安全态势、跨信任边界行动、绕过审查/影响他人。17% 的漏报率是因为分类器正确识别了危险，但无法判断先前的批准是否涵盖这次行动的实际影响范围。
